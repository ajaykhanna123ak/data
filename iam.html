The Four Pillars of IAM

Users

An identity for a single person or application with permanent credentials (passwords or access keys). By default, a new user has zero permissions.

Groups

A collection of users. Permissions are attached to the group, and all users within it inherit them, simplifying management at scale.

Policies

The "rulebook" of IAM. A JSON document defining permissions with an `Effect` (Allow/Deny), `Action`, and `Resource`.

Roles

An identity with temporary credentials that can be *assumed* by a trusted user or service. The most secure way to delegate permissions.

Advanced Governance & Security

Privileged Access Management (PAM)

PAM is a critical cybersecurity strategy focused on securing administrator and other "privileged" accounts. These accounts are the primary target for attackers.

Illustrative data showing the high percentage of breaches involving compromised privileged credentials.

IAM Policy Analyzer

An automated tool that acts as a security expert, scanning policies to find misconfigurations and unintended public access before they can be exploited.

Service Accounts

Non-human identities for applications and automation. Best practice is to assign them to IAM Roles for temporary, secure credentials.

Labels (or Tags)

Key-value metadata for organizing resources, enabling cost allocation, and automating tasks. Example:

env:production

Organization Policies

Centralized "guardrails" that restrict resource configurations (e.g., limit server locations) to enforce compliance for everyone.

IAM vs. Organization Policies

IAM Policies

Control WHO can do WHAT.

Focuses on user and service permissions.

Organization Policies

Control WHAT is allowed to EXIST.

Focuses on resource configuration and compliance guardrails.
